# Thompson Sampling (Сэмплирование Томпсона)

## 1. Основная философия (Байесовский подход)
В отличие от классических (частотных) методов вроде $\epsilon$-Greedy или UCB1, алгоритм Томпсона опирается на **Байесовскую статистику**. 

Главное отличие: истинная доходность (или CTR) рекламного места или стратегии считается не фиксированным неизвестным числом, а **случайной величиной**, которая описывается **распределением вероятностей**.

* **Широкое распределение (Fat):** Мы почти ничего не знаем о стратегии. Дисперсия огромная, неуверенность высокая.
* **Узкое распределение (Skinny):** Мы собрали много данных. Мы точно знаем, какой здесь средний доход.



---

## 2. Суть алгоритма (Как он принимает решения)
Алгоритм не использует формулы доверительных интервалов или жесткие проценты трафика. Он использует **сэмплирование (выборку)**:

1. На каждом шаге (для каждого пользователя/запроса) алгоритм обращается к распределениям всех доступных вариантов (ручек).
2. Из каждого распределения он генерирует ровно **одно случайное число (сэмпл)**.
3. Вариант, чей сэмпл оказался **максимальным**, побеждает и забирает показ.
4. После получения результата (клик/доход) распределение победителя обновляется.

**Почему это гениально:** Если вариант не изучен (широкий колокол), он может случайно выдать очень большое число и победить стабильного лидера. Так происходит органичное **исследование (Exploration)**. По мере накопления данных колокол сужается, случайные "выбросы" прекращаются, и алгоритм переходит в чистую **эксплуатацию (Exploitation)** лучшего варианта.

## 3. Математический движок: Сопряженные распределения
Для обновления знаний используется Формула Байеса:
$$P(\theta | X) \propto P(X | \theta) \cdot P(\theta)$$
*(Апостериорная вероятность пропорциональна Правдоподобию, умноженному на Априорную вероятность).*

Чтобы не считать сложные интегралы в реальном времени, используются **сопряженные распределения (Conjugate Priors)** — математический трюк, который превращает обновление модели в простое сложение чисел.

### Сценарий А: Бинарная награда (Клики: 0 или 1)
Используется пара **Bernoulli / Beta**.
* **Распределение:** Бета-распределение $Beta(\alpha, \beta)$.
* **Смысл параметров:** $\alpha$ — количество успехов (кликов), $\beta$ — количество неудач.
* **Обновление:** * Если успех ($x=1$): $\alpha_{new} = \alpha + 1$
  * Если неудача ($x=0$): $\beta_{new} = \beta + 1$



### Сценарий Б: Непрерывная награда (Доход/RPS, Время)
Используется пара **Gaussian / Gaussian** (Нормальное распределение). Это твой случай для оптимизации таймаутов и прайс-флоров!
* **Распределение:** Нормальное распределение $Normal(m, \lambda)$.
* **Смысл параметров:** $m$ — наша оценка среднего (Mean), $\lambda$ — точность нашей оценки (Precision, обратная дисперсии). Чем больше данных, тем выше $\lambda$.
* **Обновление:** Новое среднее ($m_{new}$) становится взвешенным средним между нашими старыми знаниями и новым полученным доходом. Точность ($\lambda$) увеличивается с каждым показом.

## 4. Пошаговый алгоритм в продакшене (AdTech)

1. **Инициализация:** Задать начальные параметры (Prior) для всех стратегий. Например, для RPS: $m = 0$, $variance = 0.1$ (очень широкие, неуверенные распределения).
2. **Получение запроса:** Пришел пользователь.
3. **Сэмплирование:** Для каждой стратегии $i$ генерируем случайное число $\theta_i$ из её текущего распределения.
   * *Для Гаусса в JS:* `sample = randomGaussian(mean, variance)`
4. **Выбор (Greedy):** Выбираем стратегию с максимальным $\theta_i$.
5. **Показ и сбор:** Применяем стратегию (например, ставим `price_floor`), ждем завершения аукциона, записываем результат $x$ (доход).
6. **Обновление (Update):** Пересчитываем $m$ и $variance$ (или $\alpha$ и $\beta$) для победившей стратегии с учетом нового результата $x$.

## 5. Плюсы и минусы Thompson Sampling

| Плюсы | Минусы |
| :--- | :--- |
| **Естественный баланс:** Автоматически снижает долю исследований по мере роста уверенности. Никаких хардкод-процентов. | **Сложность понимания:** Требует базового понимания теории вероятностей (сложнее объяснить бизнесу, чем 10% рандома). |
| **Адаптивность:** Идеален для отложенной обратной связи (Delayed Feedback) и батчевого обновления (как в Snowflake раз в час). | **Генерация случайных чисел:** Сэмплирование из сложных распределений (Гаусс, Гамма) требует вычислительных ресурсов или доп. функций. |
| **Эффективность:** Математически доказано, что TS имеет одну из самых низких "цен ошибок" (Regret) на длинной дистанции. | **Чувствительность к Prior:** Если задать неадекватные начальные параметры, алгоритм может долго сходиться к истине. |

## 6. Как это ложится на твою архитектуру (Edge + DWH)
Thompson Sampling идеально вписывается в вашу систему:
1. **DWH (Snowflake):** Выполняет роль "Обновителя" (Updater). Он раз в час агрегирует доходы, считает новые математические параметры `mean` и `variance` для каждого сегмента и выгружает их в статический JSON.
2. **CDN (Cloudflare Worker):** Выполняет роль "Сэмплера" (Sampler). Он читает JSON, генерирует по одному случайному числу для каждой ручки по Гауссу и моментально принимает решение, какую стратегию отдать браузеру пользователя.
